Thumbify Setup Notes
====================

Project structure:
- The React client app is inside the `client` folder.
- Run npm commands from `client` unless a later step says otherwise.

Step 1: Frontend package installation
-------------------------------------

Command:

```bash
cd client
npm install @tailwindcss/vite tailwindcss lenis lucide-react motion react-fast-marquee react-router-dom
```

Dependencies added and why they are used:

- `@tailwindcss/vite`
  - Adds Tailwind CSS v4 directly into the Vite build pipeline.
  - Used in `client/vite.config.ts` through the `tailwindcss()` Vite plugin.

- `tailwindcss`
  - Provides the utility-first CSS framework used for styling the React UI.
  - Used in `client/src/globals.css` with `@import "tailwindcss";`.

- `lenis`
  - Adds smooth scrolling behavior to the page.
  - Used in `client/src/components/LenisScroll.tsx`.

- `lucide-react`
  - Provides lightweight React icon components.
  - Used across navbar, hero, features, pricing, contact, and footer components.

- `motion`
  - Provides animation primitives for React components.
  - Used through `motion/react` for entrance animations, hover/tap animation, and interactive image tilt effects.

- `react-fast-marquee`
  - Provides a ready-made marquee/continuous scrolling component.
  - Used in the testimonial section to scroll testimonial cards horizontally.

- `react-router-dom`
  - Provides client-side routing for the React app.
  - Used in `client/src/main.tsx`, `client/src/App.tsx`, navbar links, and footer links.

Notes:
- `react` and `react-dom` are the core React runtime dependencies already required by the Vite React app.
- Vite, TypeScript types, ESLint, and React plugin packages are development tooling for running, building, and linting the client app.

Step 2: Backend package installation
------------------------------------

Command:

```bash
cd server
npm install bcrypt connect-mongo cors dotenv express express-session mongoose
npm install -D @types/bcrypt @types/cors @types/express @types/express-session @types/node nodemon ts-node tsx typescript
```

Dependencies added and why they are used:

- `express`
  - Creates the backend HTTP server and route layer.
  - Used in `server/server.ts` to initialize the app and define the health route.

- `cors`
  - Allows the frontend client to call the backend from a different origin during development.
  - Used as middleware in `server/server.ts`.

- `dotenv`
  - Loads environment variables from `.env` into `process.env`.
  - Used in `server/server.ts` through `import "dotenv/config"`.

- `mongoose`
  - Connects the app to MongoDB and defines database models/schemas.
  - Used in `server/configs/db.ts` for the database connection and in `server/models/User.ts` for the user schema.

- `bcrypt`
  - Used to hash and compare passwords securely for authentication.
  - Installed for user auth flows; not yet used in the current backend files.

- `express-session`
  - Adds session support for login state and server-managed authentication flows.
  - Used in `server/server.ts` to store login-related session data and configure session cookies.

- `connect-mongo`
  - Lets Express sessions be stored in MongoDB instead of memory.
  - Installed for persistent session storage with MongoDB-backed auth flows; not yet wired into the current session config.

Dev dependencies added and why they are used:

- `typescript`
  - Enables TypeScript compilation for the backend.
  - Used by the `build` script: `tsc`.

- `tsx`
  - Runs TypeScript files directly in development without a manual compile step.
  - Used by the `start` and `dev` scripts.

- `nodemon`
  - Restarts the backend automatically when server files change.
  - Used by the `dev` script together with `tsx`.

- `@types/express`
  - Provides TypeScript types for Express APIs.
  - Supports typed `Request` and `Response` usage in `server/server.ts`.

- `@types/cors`
  - Provides TypeScript types for the `cors` package.

- `@types/bcrypt`
  - Provides TypeScript types for `bcrypt`.
  - Useful when password hashing and comparison logic is added to auth controllers, model hooks, or service functions.

- `@types/express-session`
  - Provides TypeScript types for session middleware.

- `@types/node`
  - Provides TypeScript types for Node.js globals and runtime APIs.

- `ts-node`
  - Executes TypeScript in Node-based workflows.
  - Installed as backend tooling, but the current scripts use `tsx` instead.

Notes:
- The backend currently starts with `npm run dev` or `npm start` from the `server` folder.
- The server expects `MONGODB_URI` and `SESSION_SECRET` in `server/.env`.
- CORS is currently configured to allow `http://localhost:5173` and `http://localhost:3000` with credentials enabled.
- `bcrypt`, `@types/bcrypt`, `connect-mongo`, and `ts-node` are installed now for upcoming auth/backend work, even though they are not fully used yet in the checked files.

Step 3: Client API and auth state packages
------------------------------------------

Command:

```bash
cd client
npm install @tanstack/react-query axios react-hot-toast
```

Dependencies added and why they are used:

- `@tanstack/react-query`
  - Manages async server state, mutations, caching, and request lifecycle for the frontend.
  - Used in `client/src/App.tsx` through `QueryClient` and `QueryClientProvider`, and in `client/src/context/AuthContext.tsx` for auth mutations and user verification flows.

- `axios`
  - Provides a simple HTTP client for calling backend APIs.
  - Used in `client/src/configs/api.ts` to create a shared API client with `baseURL` and `withCredentials`.

- `react-hot-toast`
  - Shows lightweight toast notifications for success and error states.
  - Used in `client/src/context/AuthContext.tsx` for login, signup, logout, and error feedback.

Notes:
- The client API layer reads `VITE_BASE_URL` and falls back to `http://localhost:3000`.
- `axios` is configured with `withCredentials: true`, which matches the backend session-based auth setup.

Step 4: Server AI and media storage packages
--------------------------------------------

Command:

```bash
cd server
npm install @google/genai cloudinary
```

Dependencies added and why they are used:

- `@google/genai`
  - Connects the backend to Google Gemini models for AI-powered content generation.
  - Used in `server/configs/ai.ts` to create the Gemini client and in `server/controllers/ThumbnailController.ts` to generate thumbnail images.

- `cloudinary`
  - Uploads and stores generated images in a hosted media service.
  - Used in `server/controllers/ThumbnailController.ts` to upload generated thumbnail files after local image creation.

Notes:
- The backend now includes AI thumbnail generation logic in `server/controllers/ThumbnailController.ts`.
- The Gemini client expects `GEMINI_API_KEY` in the server environment.
- The thumbnail generation flow also needs Cloudinary credentials configured before uploads can work properly.
